## コミュニケーションの癖

### 返答の意味
- 「よろ」= 進めて OK（PR 作成までは OK、マージは別途確認）
- 「おk」「OK」= 確認した / 了解した
- 「マージして」= マージの明示的な承認
- 「見せて」= `open` コマンドで Finder/プレビューで開く

### 好み
- 長い説明よりも表形式や箇条書き
- 選択肢は 2〜3 個に絞る（5 個以上は多すぎる）
- メリット・デメリットを正直に伝える。忖度しない
- 質問したら端的に答える。長い前置き不要

### 堅さのレベル
- カジュアル寄り OK
- ただし院内文書・LP コピーはやや丁寧め

## 作業パターン

### ファイル変更の確認
- `git status` で予期しない変更がないか必ず確認する習慣
- `exercises.ts` や `illustrations.ts` が突然変更されていたら、curator が別ターミナルで追加した可能性が高い
- コミット前に必ず `npx tsc --noEmit` で型チェック

### LP 画像の二重配置
- LP 画像は `assets/lp/` と `public/assets/lp/` の両方に存在
- `public/assets/lp/` は Expo ビルドで自動コピーされるため Vercel プレビューで表示
- `assets/lp/` は GitHub Actions のデプロイステップでコピー
- **新画像を追加する場合は両方に配置**

### 運動数の表記変更時にチェックするファイル
- `.github/workflows/deploy-web.yml`（SEO メタタグ、構造化データ、llms.txt）
- `public/lp.html`（FAQ 構造化データ、本文テキスト）
- LP 内の特徴セクション、FAQ セクション
- 確認コマンド: `grep -r "22種\|100種" .`

### SEO 関連の更新
- `deploy-web.yml` と `public/lp.html` は同じ情報を持つ箇所がある（構造化データ等）
- 片方だけ更新して片方を忘れるミスが起きやすい
- **変更時は必ず両方を確認**

### コミット・PR
- コミットメッセージは日本語、変更内容を端的に
- Co-Authored-By: `Claude Opus 4.6 (1M context) <noreply@anthropic.com>` を付ける
- PR タイトルも日本語
- PR body には `## Summary`（箇条書き）と `## Test plan`（チェックボックス）を含める
- タスクファイル（`phase*.md`）の更新は機能変更と同じコミットか直後の別コミット

## 地雷・注意点

### 絶対に踏まない地雷
- **PR を勝手にマージ**（最大の地雷。過去に強く注意された）
- **疾患タグを提案**（SaMD リスクで却下済み。ユーザーの信頼を損なう）
- **main に直接 push**（ブランチ保護で弾かれる、そもそも試みない）
- **App Store バッジを自作**（Apple 公式 SVG を使う）
- **「理学療法士のための」と書く**（「医療・介護の現場で使える」に統一済み）
- **不要なファイル作成**（README やドキュメントを頼まれていないのに作らない）

### 避けるパターン
- 長い説明
- バックエンド・API の提案（オフラインファーストの根幹）
- 外部ライブラリの安易な追加

## 並行作業の注意

- 3 つのターミナルが同じリポジトリで作業するためコンフリクト注意
- 作業前に `git pull` で最新を取得
- curator がデータを追加した直後に app-developer がコミットするパターンが多い
- curator が型定義の拡張を求めた場合、`src/types/exercise.ts` は app-developer が実装
